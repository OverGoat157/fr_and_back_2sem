## Краткое описание

### №19 — PostgreSQL
- Express + Sequelize + PostgreSQL.
- Сущность `User`: `id`, `first_name`, `last_name`, `age`, `created_at`, `updated_at`.
- Маршруты `/api/users` (POST, GET), `/api/users/:id` (GET, PATCH, DELETE).

### №20 — MongoDB
- Express + Mongoose + MongoDB.
- Та же сущность и тот же набор маршрутов, но данные хранятся в документной БД.

### №21 — Redis caching
- К приложению из практики №11 добавлен Redis.
- Кэшируются маршруты `GET /api/users` и `GET /api/users/:id` (TTL 1 мин), `GET /api/products` и `GET /api/products/:id` (TTL 10 мин).
- При изменении/удалении сущностей соответствующие ключи кэша удаляются.
- В ответе указано поле `source: "server" | "cache"`.

### №22 — Балансировка нагрузки
- Три backend-сервера на Node.js (порты 3000, 3001, 3002 — последний резервный).
- Конфиг `nginx.conf` с `max_fails=2 fail_timeout=30s` и backup-сервером.
- Альтернативный конфиг `haproxy.cfg` с health-checks и round-robin.

### №23 — Docker Compose
- Каждый компонент системы из практики №22 — отдельный Docker-контейнер.
- `Dockerfile` для backend-сервиса, `nginx.conf` использует имена сервисов из `docker-compose.yml`.
- Запуск всего стека одной командой `docker compose up --build`.

## Тестирование

Каждая практика содержит свой `README.md` c командами установки, запуска и проверки. Перед сдачей нужно:

1. Запустить каждое приложение и убедиться, что все маршруты работают.
2. Проверить кэш (практика 21): первый ответ — `source: "server"`, повторный — `source: "cache"`.
3. Проверить балансировку (практики 22 и 23): повторные запросы должны поочерёдно приходить от разных backend.
4. Проверить отказоустойчивость: остановка одного backend не ломает работу системы.
